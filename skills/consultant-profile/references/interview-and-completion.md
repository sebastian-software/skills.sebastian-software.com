# Interview and Completion Mode

Use this reference when a consultant profile is incomplete, project facts are unclear, sources disagree, or the user wants the agent to interview them before rewriting.

The goal is to build a reliable evidence base before shaping copy. Do not make the user answer everything at once. Ask the few questions that unblock the next useful improvement.

## When to Use This Mode

Use interview and completion mode when:

- the current website only mentions some projects briefly
- older projects exist on an old homepage, LinkedIn, a PDF CV, proposals, notes, or archived bios
- the project list has client names but not enough role, scope, outcome, or context
- LinkedIn recommendations or references contain useful external perspective
- the profile claims are plausible but need support
- the user says they are not sure what belongs in the profile
- the user wants a structured intake before rewriting

## Source Inventory

Start by listing available sources and what each source can contribute.

Common sources:

- current website profile
- old website profile
- LinkedIn profile
- LinkedIn recommendations
- XING profile
- PDF consultant profiles or CVs
- old proposals, bios, speaker notes, project lists
- client references/testimonials
- case studies, blog posts, talks, podcasts, GitHub, open-source work
- private user memory or interview answers

For each source, capture:

- source name and URL/path if available
- date or freshness if known
- project facts it contains
- wording or perception signals it contains
- conflicts with other sources
- permission constraints

## Project Record Schema

Normalize each project into a structured record before writing.

Use this schema mentally or in output:

- **Client / organization** - exact name, anonymized name, or client type
- **Industry** - banking, telecom, retail/ecommerce, industrial/B2B, SaaS, public sector, etc.
- **Period / duration** - exact if known, approximate if not
- **Role** - consultant role, seniority, team position, ownership
- **Situation** - why the project existed
- **Scope** - product, system, users, regions, brands, teams, stakeholders
- **Responsibilities** - what the consultant personally owned
- **Deliverables** - architecture, implementation, strategy, migration, design system, audit, roadmap, etc.
- **Technologies / methods** - only where relevant
- **Outcomes** - metrics, shipped artifacts, decisions enabled, quality, speed, adoption, risk reduction
- **Constraints** - regulated, high-traffic, multi-brand, multilingual, legacy, stakeholder complexity
- **Narrative links** - what earlier experience fed into it, what later work built on it
- **Prominence score** - signature, supporting, archive, or source-only
- **Open questions** - missing facts or claims to verify
- **Permission** - public, anonymized, internal only, unknown

## Completeness Pass

Before rewriting, classify every project and source:

- **Complete enough** - can be used in selected highlights or project entries.
- **Promising but incomplete** - likely important, needs targeted questions.
- **Mention-only** - name/client exists but not enough context yet.
- **Reference-only** - useful as proof or perception signal, not a project entry.
- **Archive-only** - keep for completeness but probably not prominent.
- **Needs permission** - do not publish client name, quote, metric, or outcome until verified.

Then produce a gap map:

- Missing high-signal projects
- Missing outcomes or metrics
- Missing role clarity
- Missing industry/client context
- Missing narrative links
- Missing proof for positive claims
- Duplicates or conflicts between sources
- Client-name and confidentiality risks

## Interview Mode

Ask concise, prioritized question batches. Prefer 3-7 questions at a time. Do not ask every possible question.

Good interview behavior:

- Start with the highest-signal missing project or claim.
- Ask questions the user can answer from memory.
- Offer "unknown / skip / anonymize" as valid answers.
- Preserve uncertainty instead of forcing precision.
- After each answer batch, update the project record and ask the next useful batch.

### First Intake Questions

Use these when starting from incomplete project data:

1. Which projects must not be missing from the final profile, even if they are older?
2. Which client names can be public, anonymized, or not used at all?
3. Which projects best prove what you want to be hired for next?
4. Are there old sources I should compare, such as the previous homepage, LinkedIn, or PDFs?
5. Are there LinkedIn recommendations or references that describe your work especially well?

### Project Clarification Questions

Use these for one project at a time:

1. What was the actual business or product problem?
2. What did you personally own or lead?
3. Who were the main stakeholders or teams?
4. What shipped, changed, improved, or became possible?
5. What made the environment hard: scale, regulation, legacy, language, brands, politics, time pressure?
6. Which earlier experience helped you here?
7. Did this project later influence another project or capability?

### Outcome Questions

Ask when the project feels too task-oriented:

1. Was there a measurable result?
2. If not measurable, what qualified outcome is safe to say?
3. Did the work reduce risk, speed up delivery, improve maintainability, clarify strategy, or enable decisions?
4. Did someone else describe the impact in a reference, recommendation, or testimonial?

### Narrative Questions

Ask when the project list feels disconnected:

1. Which theme keeps coming back across projects?
2. Where did you first learn a pattern that later became important?
3. Which project changed your seniority or the kind of responsibility clients trusted you with?
4. Which industries or operating contexts did you return to?
5. What would a client miss if they saw only the latest three projects?

## Using LinkedIn Recommendations and References

Do not paste long recommendations into the main profile unless the output format explicitly calls for testimonials. Long references usually work better as source material than as body copy.

Extract these signals instead:

- **Perceived strengths** - what others repeatedly say the consultant is good at
- **Working style** - calm, pragmatic, clear, fast, reliable, collaborative, strategic, hands-on
- **Trust signals** - senior stakeholders trusted them, difficult work was assigned, teams wanted to work with them again
- **Impact language** - what others say changed because of the work
- **Differentiators** - unusual combinations such as technical depth plus product judgment
- **Exact phrases** - short phrases that can inspire profile language, if permission allows

Classify each reference:

- direct testimonial candidate
- proof for a capability claim
- working-style signal
- project impact clue
- narrative-arc clue
- not relevant for public profile

## Where References Belong

Use references differently by channel:

- **LinkedIn** - keep recommendations on LinkedIn; summarize recurring themes in About or Experience, and point readers to recommendations only if useful.
- **Website** - use one or two short testimonial excerpts only when they support the page's positioning and permission is clear.
- **PDF consultant profile** - usually summarize proof signals; include full references only in a separate appendix or "References available" section.
- **Proposal bio** - use the most relevant reference angle for the buyer's situation.
- **Internal source of truth** - store full recommendation text, source, date, author, relationship, permission, and extracted signals.

Avoid turning the profile into a testimonial wall. References are evidence, not the structure.

## Recommended Output for Completion Mode

Return:

1. **Source Inventory** - sources reviewed or requested.
2. **Project Inventory** - known projects, missing projects, duplicates, and likely high-signal gaps.
3. **Completeness Table** - complete, incomplete, mention-only, reference-only, archive-only, needs permission.
4. **Interview Questions** - the next 3-7 highest-value questions.
5. **Reference Signal Map** - themes extracted from recommendations/references.
6. **Next Profile Update** - what can be safely improved now vs what should wait for answers.

## Guardrails

- Do not invent missing project details.
- Do not publish private recommendations without permission.
- Do not quote long references unless explicitly requested and allowed.
- Do not let data collection block all progress; improve what is supported, then mark the rest as open.
- Do not over-interview. Ask enough to unblock the next profile improvement.

The best interview mode feels like a senior editor helping the user remember and organize the evidence they already have.
